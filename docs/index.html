01 — The Problem

Models rot
in silence.

You train a model, it hits 92% accuracy, you ship it. Six months later it's making bad predictions — but nobody noticed because there were no errors, just quietly wrong outputs.

01
Training Distribution
Your model learns patterns from historical data. It builds a "mental model" of what the world looks like.
02
The World Changes
Customer behaviour shifts. Prices inflate. New products launch. The world your model learned no longer exists.
03
Silent Degradation
No exceptions are thrown. The model keeps predicting. But accuracy silently collapses — often undetected for months.
04
Drift Detection
We solve this by statistically comparing what the model sees today vs. what it was trained on — and alerting when they diverge.

03 — Drift Detection

Three ways
data breaks.

We implemented three industry-standard statistical tests. Each catches a different failure mode — together they form a complete drift detection system.

Test Feature Type What It Measures Threshold Status Example
Kolmogorov–Smirnov
scipy.stats.ks_2samp
 Numerical Max distance between two empirical CDFs. KS=0 means identical distributions, KS=1 means completely different. > 0.10 ALERT: tenure 0.20
Chi-Square Test
scipy.stats.chi2_contingency
 Categorical Compares observed vs expected category frequencies. High chi² or low p-value → the category mix has shifted. p < 0.05 ALERT: Contract drifted
Population Stability Index
Custom implementation
 Numerical Credit scoring standard. Bins both distributions, compares proportions. PSI <0.1 stable, 0.1–0.2 moderate, >0.2 significant. > 0.20 ALERT: MonthlyCharges 0.67
PSI Formula
# Population Stability Index — the industry-standard drift magnitude # PSI = Σ (actual% − expected%) × ln(actual% / expected%) def compute_psi(reference, current, bins=10): breakpoints = np.percentile(reference, np.linspace(0, 100, bins+1)) ref_pct = np.histogram(reference, bins=breakpoints)[0] / len(reference) cur_pct = np.histogram(current, bins=breakpoints)[0] / len(current) ref_pct = ref_pct.clip(1e-6) # avoid log(0) cur_pct = cur_pct.clip(1e-6) return np.sum((cur_pct - ref_pct) * np.log(cur_pct / ref_pct))

04 — Pipeline Walkthrough

Five phases,
fully automated.

Run python main.py and all five phases execute sequentially. Here's what happens under the hood.

1
Data Loading and Preprocessing
We use the Telco Customer Churn dataset — 7,000 rows, 19 features (mix of numerical and categorical). The dataset is auto-generated synthetically if not present. We clean types, handle missing values, and split 85/15 into train/validation while preserving class balance via stratification.
X_train, X_val, y_train, y_val = train_test_split( X, y, test_size=0.15, stratify=y, random_state=42 )
2
Baseline Model Training
CatBoostClassifier is trained with early stopping on the validation set. It handles categorical features natively — no one-hot encoding required. We track F1-macro as our primary metric because it's robust under class imbalance (our dataset has ~60% churners).
model = CatBoostClassifier( iterations=500, learning_rate=0.05, depth=6, early_stopping_rounds=50 ) # Baseline F1-macro: ~0.68
3
Drift Simulation
We create 6 production batches with escalating drift (drift_level 0.0 to 1.0). Three types of drift are injected: numerical shift (tenure drops 15 months, MonthlyCharges rises $25), categorical skew (Contract shifts to Month-to-month, PaymentMethod toward Electronic check), and label drift (churn rate rises from 60% to 80%).
# Batch 1: drift_level=0.00 — churn_rate=66% — stable # Batch 4: drift_level=0.60 — churn_rate=75% — moderate drift # Batch 6: drift_level=1.00 — churn_rate=81% — severe drift
4
Drift Detection Per Batch
For each batch we run all three statistical tests across all features, then compute a unified drift_score per feature (0–1). The overall_drift_flag triggers if more than 30% of features are flagged OR max PSI exceeds 0.20. Performance drift triggers if F1 drops more than 0.05 from baseline.
overall_drift = (frac_drifted > 0.30) or (max_psi > 0.20) perf_drift = (baseline_f1 - current_f1) > 0.05 should_retrain = overall_drift or perf_drift
5
Dashboard and Audit Log
All results are serialised to data/pipeline_results.pkl and read by the Streamlit dashboard. Every retraining event is logged to logs/retrain_log.csv with timestamp, reason, pre/post F1 — a full audit trail.
05 — Auto-Retraining

The model fixes
itself.

When drift exceeds thresholds, the system retrains automatically on the latest batch, replaces the saved model, and logs the event — no human intervention needed.

Why CatBoost?
Handles categorical features natively — no one-hot encoding needed. Strong baseline performance on tabular data. Industry standard at Yandex, Sber, and many fintech companies. Native early stopping.
Why PSI + KS + Chi2?
KS is sensitive to shape changes. PSI gives a magnitude score (0–1) with industry-standard thresholds from credit scoring. Chi2 handles the categorical side. Together they catch every drift pattern.
07 — Project Summary

What this
system does.

A concise overview of what the ML Drift Monitoring System detects, how it responds, and why it matters in production environments.

Continuous Monitoring
The system watches incoming data batches in real time, comparing them against the original training distribution to catch changes before they impact users.
Statistical Rigor
Three complementary tests — KS, Chi-Square, and PSI — cover both numerical and categorical features, leaving no drift pattern undetected.
Automated Recovery
When drift is confirmed, the system retrains the model automatically on fresh data and replaces the old version — no manual intervention required.
Full Transparency
Every decision is logged. The dashboard shows exactly which features drifted, by how much, and what action was taken — giving teams full visibility at all times.
